## 三、项目配置详解

### 3.1 package.json

```json
{
  "name": "mozhi-academy",
  "version": "0.1.0",
  "private": true,
  "type": "module"
}
```

**关键决策**：

- **`"type": "module"`** — 全项目使用 ESM 模块系统。tsx 和 Vite 都能完美处理 ESM，避免了 CJS/ESM 混用的坑。代价是不能直接使用 `__dirname`，需要用 `process.cwd()` 或 `import.meta.url` 替代。

**依赖说明**：

| 依赖 | 用途 |
|------|------|
| `express` ^4.21.0 | API 服务器 |
| `better-sqlite3` ^11.7.0 | SQLite 数据库驱动 |
| `bcryptjs` ^2.4.3 | 密码哈希（选纯 JS 版而非 `bcrypt`，避免 native 编译问题）— 认证已禁用 |
| `jsonwebtoken` ^9.0.2 | JWT token 生成/验证 — 认证已禁用 |
| `@anthropic-ai/sdk` ^0.39.0 | Anthropic Claude API |
| `openai` ^4.77.0 | OpenAI GPT API |
| `cors` ^2.8.5 | 跨域请求支持 |
| `dotenv` ^16.4.7 | 环境变量加载 |

| 开发依赖 | 用途 |
|----------|------|
| `tsx` ^4.19.0 | TypeScript 直接执行（替代 ts-node，更快，ESM 更好） |
| `vite` ^6.0.0 | 前端构建 + 开发服务器 |

**npm 脚本**：

### 3.2 tsconfig.json

```json
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "jsx": "react-jsx",
    "strict": true,
    "paths": { "@/*": ["./src/*"] }
  },
  "include": ["src", "server"]
}
```

**关键决策**：

- **单一 tsconfig** — 前后端共用一份配置。`"moduleResolution": "bundler"` 让 tsx 和 Vite 都能正确解析模块。`"noEmit": true` 表示 TypeScript 只做类型检查，不做编译（tsx/Vite 各自处理编译）。
- **`"paths": {"@/*": ["./src/*"]}`** — 前端代码可以用 `@/types/index` 代替 `../../types/index`，Vite 通过 `vite.config.ts` 的 alias 配置解析。
- **严格模式** — `strict: true` + `noUnusedLocals` + `noUnusedParameters`，确保代码质量。

### 3.3 vite.config.ts

```ts
export default defineConfig({
  plugins: [react()],
  resolve: { alias: { '@': path.resolve(__dirname, './src') } },
  server: {
    port: 3000,
    proxy: { '/api': { target: 'http://localhost:3001', changeOrigin: true } },
  },
  build: { outDir: 'dist' },
});
```

**工作原理**：

- 开发时，浏览器访问 `http://localhost:3000`（Vite）
- Vite 拦截所有 `/api/*` 请求，自动转发到 `http://localhost:3001`（Express）
- 前端代码无需关心后端端口，直接 `fetch('/api/...')` 即可
- `'X-Accel-Buffering': 'no'` 在 SSE helper 中设置，防止 nginx 等反向代理缓冲流式响应

移除了 Next.js 相关条目（`/.next/`、`/out/`、`next-env.d.ts`、`.vercel`），新增：

## 四、后端模块详解

### 4.1 server/index.ts — Express 入口

```
dotenv 加载 → Express 实例 → CORS → JSON 解析 → Auth 中间件（已禁用，直接放行） → 路由挂载 → 静态文件 → 监听端口
```

**启动流程**：

1. `import 'dotenv/config'` — 最先加载 `.env` 文件中的环境变量
2. `import './db'` — 初始化数据库（副作用导入：建表 + 初始化 settings 单例行）
3. 挂载中间件：`cors()` → `express.json()` → `authMiddleware`（已禁用，直接放行）
4. 挂载路由：`/api/auth` → authRoutes
5. 生产模式：从 `dist/` 提供静态文件 + SPA fallback

> **注意**：认证中间件已禁用（学习项目无需认证），`authMiddleware` 直接调用 `next()` 放行所有请求。

### 4.2 server/db.ts — 数据库

**设计决策**：

- **文件位置**：`data/mozhi.db`（相对于项目根目录，`process.cwd()` 解析）
- **自动建目录**：首次运行自动创建 `data/` 目录
- **WAL 模式**：`journal_mode = WAL` 提升并发读性能
- **外键约束**：`foreign_keys = ON`，确保数据完整性
- **单例行模式**：`auth_config` 和 `settings` 表用 `CHECK (id = 1)` 约束为单例（只有一个用户的应用）
- **settings 初始化**：建表后自动插入 `id=1` 的空行，避免后续查询时需要判断行是否存在

**14 张表结构一览**：

### 4.3 server/middleware/auth.ts — 认证中间件（已禁用）

> **⚠️ 认证已禁用**：本项目为学习项目，不启用真实认证。中间件直接调用 `next()` 放行所有请求。

**导出**：

1. `authMiddleware(req, res, next)` — Express 中间件函数（已禁用，直接 `next()`）

**原始认证流程（已禁用）**：

```
请求进来
  │
  ├─ 路径以 /auth 开头？ ──是──→ next()（跳过认证）
  ├─ 路径是 /health？    ──是──→ next()（跳过认证）

```

  // 认证已禁用，req.user 不会被设置，忽略即可

```

### 4.4 server/services/crypto.ts — AES-256-GCM 加解密

**用途**：加密用户的 API Key 存入数据库。

**加密流程**：

```
明文 API Key
  → scrypt(密钥种子, "mozhi-salt-v1") → 32 字节主密钥
  → randomBytes(16) → IV（初始化向量）
  → AES-256-GCM 加密(明文, 主密钥, IV)
  → 输出格式: "iv_hex:authTag_hex:ciphertext_hex"
```

**解密流程**：反向操作，从 `:` 分隔的字符串中提取 IV、authTag、密文，然后解密。

**安全说明**：
- 每次加密使用随机 IV，相同明文每次产生不同密文
- GCM 模式自带认证标签（authTag），能检测数据是否被篡改
- 开发环境使用默认密钥，生产环境应设置 `ENCRYPTION_KEY` 环境变量

**如何使用**：

```typescript
import { encrypt, decrypt } from '../services/crypto';

// 存储 API Key
const encrypted = encrypt('sk-ant-xxxxx');
db.prepare('UPDATE settings SET anthropic_key_enc = ? WHERE id = 1').run(encrypted);

// 读取 API Key
const row = db.prepare('SELECT anthropic_key_enc FROM settings WHERE id = 1').get();
const apiKey = decrypt(row.anthropic_key_enc);
```

### 4.5 server/services/ai.ts — AI 双 Provider 服务

**设计架构**：

```
调用方
  │
  ├─ chat(systemPrompt, messages)         → 返回完整文本
  └─ chatStream(systemPrompt, messages)   → AsyncGenerator<string> 逐块返回
      │
      └─ getAISettings()
           从 settings 表读取 → decrypt 解密 → 确定使用哪个 Provider
              │
              ├─ preferred_provider === 'anthropic' && key 存在 → Anthropic SDK
              └─ preferred_provider === 'openai' && key 存在   → OpenAI SDK
```

**接口说明**：

```typescript
import { chat, chatStream, ChatMessage } from '../services/ai';

// 非流式调用（适用于内容生成：大纲、笔记等）
const response: string = await chat(
  '你是一位课程设计专家',
  [
    { role: 'user', content: '帮我设计 Python 入门课程' }
  ]
);

// 流式调用（适用于 AI 对话：逐字输出）
try {
  for await (const chunk of chatStream(
    '你是一位 AI 导师',
    [
      { role: 'user', content: '什么是递归？' }
    ]
  )) {
    // chunk 是一小段文本，如 "递归"
    // 通过 SSE 发送给前端
    sendSSEChunk(res, chunk);
  }
} catch (err) {
  sendSSEError(res, (err as Error).message);
}
```

**Provider 切换逻辑**：
1. 读取 `settings.preferred_provider`（默认 `'anthropic'`）
2. 如果选了 `openai` 且 OpenAI key 存在 → 用 OpenAI
3. 否则 → 用 Anthropic（如果 key 存在）
4. 都没有 key → 抛出错误 `"请先在设置中配置 API Key"`

**两个 Provider 的差异处理**：
- **Anthropic**：`system` 是单独参数，messages 只有 `user` 和 `assistant`
- **OpenAI**：`system` 作为 messages 数组第一条 `{ role: 'system', content } `

**注意**：每次调用都从数据库读取 settings 并创建新的 SDK client 实例。这是故意的设计——因为用户可能随时更改 API Key，缓存会导致用过期的 key。

### 4.6 server/services/terminal.ts — 代码执行沙箱

**执行流程**：

```
executeCode(code, language, timeout=10000)
  │
  ├─ 生成临时文件: /tmp/mozhi-{timestamp}.{ext}
  ├─ fs.writeFileSync 写入代码
  │
  ├─ spawn(child_process)
  │   ├─ python → python3 /tmp/mozhi-xxx.py
  │   ├─ js     → node /tmp/mozhi-xxx.js
  │   └─ ts     → npx tsx /tmp/mozhi-xxx.ts
  │
  ├─ 收集 stdout + stderr
  ├─ 等待进程结束（超时自动 kill）
  │
  └─ finally: 删除临时文件
```

**返回值**：

```typescript
interface ExecutionResult {
  stdout: string;     // 标准输出
  stderr: string;     // 标准错误（包括编译错误、运行时错误）
  exitCode: number;   // 退出码（0 = 成功）
  timedOut: boolean;  // 是否超时
}
```

**当前支持的语言**：Python、JavaScript、TypeScript。其他语言返回 `"不支持的语言"` 错误。如需扩展，在 `EXTENSIONS` 和 `COMMANDS` 映射中添加即可。

### 4.7 server/services/generator.ts — 内容生成器

**架构**：Generator = AI 服务 + 提示词模板 + JSON 解析

```
generateSyllabus(context) / generateLab(context) / generateProject(context) / generateNotes(context)
  │
  ├─ 调用对应的 prompts/*.ts → 得到 { system, user }
  ├─ 调用 ai.chat(system, [{ role: 'user', content: user }])
  ├─ AI 返回文本
  └─ extractJSON(response) → 解析 JSON（兼容 ```json 代码块包裹）
```

**四个生成函数**：

| 函数 | 输入 | 输出 | 返回类型 |
|------|------|------|----------|
| `generateSyllabus(ctx)` | 课程标题、描述、风格、格式、周数 | `{ weeks: [...] }` | JSON 对象 |
| `generateLab(ctx)` | 课程名、周主题、描述、语言 | lab 对象（含 starter_code、test_cases） | JSON 对象 |
| `generateProject(ctx)` | 课程名、周主题、描述、语言 | project 对象（含 milestones、starter_code） | JSON 对象 |
| `generateNotes(ctx)` | 课程名、周主题、描述、已有笔记 | Markdown 文本 | string |

**`extractJSON` 的鲁棒性处理**：AI 有时会用 ` ```json ... ``` ` 包裹 JSON 输出。`extractJSON` 先尝试匹配代码块，匹配不到则直接 `JSON.parse`。

**如何使用**（给队友的参考）：

```typescript
import { generateSyllabus, generateLab, generateNotes } from '../services/generator';

// 队友 B：创建课程后生成大纲
router.post('/api/courses', async (req, res) => {
  // 1. 插入课程到数据库
  const result = db.prepare('INSERT INTO courses ...').run(...);

  // 2. AI 生成大纲
  const syllabusData = await generateSyllabus({
    title: 'Python 入门',
    description: '从零开始学 Python',
    style: 'academic',
    format: 'cs61b',
    weeks: 12,
  });

  // 3. 批量插入 syllabus 表
  const insertWeek = db.prepare(
    'INSERT INTO syllabus (course_id, week_number, topic, description) VALUES (?, ?, ?, ?)'
  );
  const insertAll = db.transaction(() => {
    for (const week of syllabusData.weeks) {
      insertWeek.run(result.lastInsertRowid, week.week_number, week.topic, week.description);
    }
  });
  insertAll();
});

// 队友 C：生成笔记
router.post('/api/courses/:id/notes/generate', async (req, res) => {
  const markdown = await generateNotes({
    courseTitle: 'Python 入门',
    weekTopic: '函数与模块',
    weekDescription: '学习如何定义函数、使用模块',
  });
  // 保存 markdown 到 topic_notes 表
});
```

### 4.8 server/helpers/sse.ts — SSE 流式响应工具

**五个函数**：

| 函数 | 作用 | 用法 |
|------|------|------|
| `initSSE(res)` | 设置 SSE 响应头，开启流式连接 | 请求开始时调用 |
| `sendSSE(res, event, data)` | 发送一个命名事件 | 通用事件发送 |
| `sendSSEChunk(res, text)` | 发送文本增量 | AI 流式输出时，每收到一块就调用 |
| `endSSE(res)` | 发送 `done` 事件并关闭连接 | 流式输出结束时调用 |
| `sendSSEError(res, message)` | 发送错误事件并关闭连接 | 出错时调用 |

**SSE 事件格式**（前端解析用）：

```
event: delta
data: {"text":"你好"}

event: delta
data: {"text":"，我是"}

event: done
data: [DONE]
```

**如何使用**（给队友 C 的参考）：

```typescript
import { initSSE, sendSSEChunk, endSSE, sendSSEError } from '../helpers/sse';
import { chatStream } from '../services/ai';

router.post('/api/chat', async (req, res) => {
  const { systemPrompt, messages } = req.body;

  initSSE(res);

  try {
    for await (const chunk of chatStream(systemPrompt, messages)) {
      sendSSEChunk(res, chunk);
    }
    endSSE(res);
  } catch (err) {
    sendSSEError(res, (err as Error).message);
  }
});
```

### 4.9 server/prompts/ — 提示词模板

5 个模块，每个导出一个函数，接收上下文参数，返回 `{ system: string, user: string }`（除了 chat.ts 直接返回 system prompt 字符串）。

| 文件 | 导出函数 | 参数 | 说明 |
|------|----------|------|------|
| `syllabus.ts` | `getSyllabusPrompt(ctx)` | title, description, style, format, weeks? | 生成 N 周课程大纲 |
| `labs.ts` | `getLabsPrompt(ctx)` | courseTitle, weekTopic, weekDescription, language? | 生成一个编程实验 |
| `projects.ts` | `getProjectsPrompt(ctx)` | courseTitle, weekTopic, weekDescription, language? | 生成一个项目（含里程碑） |
| `notes.ts` | `getNotesPrompt(ctx)` | courseTitle, weekTopic, weekDescription, existingNotes? | 生成章节学习笔记 |
| `chat.ts` | `getChatSystemPrompt(ctx)` | courseTitle, courseDescription?, mode, weekTopic? | AI 对话的 System Prompt |

**chat.ts 的 4 种对话模式**：

| 模式 | 值 | 行为 |
|------|----|------|
| 导师模式 | `tutor` | 直接回答问题，解释清晰 |
| 苏格拉底模式 | `socratic` | 不给答案，通过提问引导思考 |
| 考试模式 | `exam` | 出题考查，评分并解释 |
| 调试模式 | `debug` | 帮助调试代码，引导找 bug |
